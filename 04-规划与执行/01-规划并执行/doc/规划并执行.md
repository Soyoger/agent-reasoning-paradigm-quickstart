# 规划并执行（Plan-and-Execute）：先想清楚，再动手做

## 一句话结论

规划并执行是把任务拆成两个阶段：**先用 Planner 生成可执行计划，再由 Executor 按步骤调用工具落地**。它比单轮 ReAct 更稳定、可控，尤其适合多步骤任务。

---

## 1. Why：背景、痛点与目标

### 1.1 背景
- 在复杂任务中（多工具、多约束、多步骤），模型如果边想边做，容易中途跑偏
- 企业场景强调可追踪、可回放、可审计，要求每一步都有明确输入输出

### 1.2 痛点
- **短视决策**：只看下一步，不看全局
- **重复调用**：同类工具反复查，成本和延迟上升
- **失败扩散**：早期一步错误会传递到后续所有步骤
- **不可复盘**：缺少结构化计划，问题排查困难

### 1.3 目标指标
- **成功率**：任务闭环成功率提升
- **成本**：减少无效工具调用次数
- **时延**：减少来回推理轮数
- **可解释性**：计划与执行日志可审计

---

## 2. What：核心概念与边界

### 2.1 核心定义
- **Planner**：把用户目标转成结构化步骤（Step List）
- **Executor**：按步骤执行工具，写回结果
- **Replanner（可选）**：当某一步失败或环境变化时，局部重规划

### 2.2 系统边界
- 输入：用户目标、可用工具清单、约束条件（预算、时延、权限）
- 输出：最终答案 + 计划步骤 + 执行轨迹
- 不负责：工具底层能力本身（如地图引擎、数据库性能）

### 2.3 与 ReAct 区别
| 维度 | ReAct | Plan-and-Execute |
|------|-------|------------------|
| 决策方式 | 每轮即兴决策 | 先全局规划再执行 |
| 稳定性 | 中等 | 高 |
| 可审计性 | 一般 | 强 |
| 多步骤任务 | 可做但易漂移 | 更适配 |

---

## 3. How：原理、流程与实现

### 3.1 基础流程
1. 接收用户目标（Goal）
2. Planner 输出结构化计划（JSON）
3. Executor 逐步执行工具
4. 收集每步输出并聚合最终答案
5. 失败时触发重试或重规划

### 3.2 计划结构示例

```json
{
  "goal": "给出今天下班路线建议",
  "steps": [
    {"id": 1, "tool": "weather", "params": {"city": "上海"}, "expected": "天气信息"},
    {"id": 2, "tool": "traffic", "params": {"origin": "陆家嘴", "destination": "虹桥"}, "expected": "拥堵信息"},
    {"id": 3, "tool": "eta", "params": {"distance_km": 22, "congestion": "heavy"}, "expected": "预计到达时间"}
  ]
}
```

### 3.3 执行策略
| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 串行执行 | 严格按依赖顺序执行 | 步骤间强依赖 |
| 并行执行 | 无依赖步骤并行执行 | 查询类任务 |
| 局部重规划 | 单步失败后替换后续子计划 | 动态环境变化 |

### 3.4 成本与收益估算公式

设任务包含 \(n\) 个步骤，每步成功概率为 \(p_i\)，单步平均成本为 \(c_i\)。

任务成功概率近似为：
\[
P_{success} \approx \prod_{i=1}^{n} p_i
\]

总成本为：
\[
C_{total} = \sum_{i=1}^{n} c_i + C_{replan}
\]

结论：计划越清晰，通常 \(p_i\) 越高、无效步骤越少，综合成功率与成本更优。

## 4. 技术选型对比

| 方案 | 优点 | 缺点 | 复杂度 | 适用场景 |
|------|------|------|--------|---------|
| ReAct | 上手快，灵活 | 易漂移，复盘弱 | 低 | 简单任务 |
| Plan-and-Execute | 稳定、可审计 | 设计成本更高 | 中 | 多步骤业务流程 |
| Plan+ReAct 混合 | 稳定与灵活平衡 | 编排复杂 | 高 | 高复杂动态任务 |

选择依据：如果任务步骤 >= 3 且对可控性要求高，优先 Plan-and-Execute。

---

## 5. 优化与改进（多层次）

### 5.1 工程层优化：步骤缓存
- 收益：减少重复工具调用，降低成本与时延
- 代价：需要缓存一致性策略
- 适用：查询类步骤重复率高的场景
- 验证：命中率、端到端耗时下降

### 5.2 调度层优化：依赖图并行
- 收益：缩短总耗时
- 代价：调度器复杂度上升
- 适用：有独立子任务的计划
- 验证：关键路径耗时、并行度指标

### 5.3 推理层优化：失败局部重规划
- 收益：避免全量重跑
- 代价：需要失败分类与恢复逻辑
- 适用：外部工具偶发失败
- 验证：重跑比例、恢复成功率

## 6. 场景适配与风险

### 6.1 适用场景
- 智能座舱多约束路线规划
- 复杂工单自动处置
- 多数据源融合分析

### 6.2 不适用场景
- 一步可答的问题（直接问答）
- 极低时延、超短链路任务

### 6.3 风险清单与缓解
| 风险 | 表现 | 缓解措施 |
|------|------|---------|
| 计划质量差 | 后续全错 | 引入计划校验器 |
| 工具不稳定 | 执行中断 | 重试 + 降级工具 |
| 成本超预算 | 调用次数过多 | 步骤预算与熔断 |

## 7. 智能座舱示例（映射 Why/What/How）

### 问题
“下雨天、晚高峰，从公司回家怎么走更快更稳？”

### 计划
1. 查询天气（是否下雨）
2. 查询实时路况（拥堵等级）
3. 估算 ETA（考虑天气+拥堵）
4. 输出路线建议与备选方案

### 执行结果（示例）
- 天气：中雨
- 路况：主路重度拥堵
- ETA：主路 65 分钟，绕行 48 分钟
- 建议：优先绕行路线，若 10 分钟后拥堵下降再切回主路

结论句：Plan-and-Execute 在该场景中能把“信息收集—评估—决策”分层解耦，降低即时决策误判。

## 8. 面试官追问清单（含回答要点）

### Q1：为什么不用纯 ReAct？
- ReAct 在多步骤场景容易漂移
- Plan-and-Execute 更可控、可复盘
- 对生产系统审计与稳定性更友好

### Q2：执行过程中某一步失败怎么办？
- 先重试（幂等工具）
- 重试失败触发局部重规划
- 必要时降级到兜底策略并返回不确定性说明

### Q3：如何验收这个系统？
- 指标：任务成功率、平均调用次数、P95时延、失败恢复率
- 方法：离线回放 + 在线灰度 + 回滚机制

---

## 9. 总结提纲（可背诵）

1. Plan-and-Execute = 先规划再执行，适合多步骤任务  
2. Planner 产出结构化计划，Executor 负责落地  
3. 相比 ReAct，更稳定、更可审计  
4. 成功率近似由各步骤成功率乘积决定  
5. 关键优化：缓存、并行调度、局部重规划  
6. 风险在计划质量与工具稳定性，需校验与降级  
7. 智能座舱路线规划是典型落地场景  
8. 验收看成功率、时延、成本与恢复能力
