# 函数调用/工具调用：生产级工具调用标准

## 一句话结论

函数调用（Function Calling，也称 Tool Calling）是 OpenAI 等厂商提供的**原生 API 能力**，让模型以标准 JSON 格式返回工具调用请求，是目前生产环境中最常用的工具调用方案。

---

## 1. Why：为什么需要原生 Function Calling

### 1.1 之前方案的问题

| 方案 | 问题 |
|------|------|
| **Prompt 硬抠** | 让模型输出 `[Search(...)`，然后用正则解析，不稳定 |
| **微调（ToolFormer）** | 成本高，换工具要重训 |
| **ReAct** | 灵活但依赖 Prompt，不同模型表现差异大 |

### 1.2 目标
提供**标准化、可靠、开箱即用**的工具调用能力。

---

## 2. What：Function Calling 核心概念

### 2.1 API 交互流程

```
┌──────────┐                         ┌─────────┐
│   用户   │                         │  模型   │
└────┬─────┘                         └────┬────┘
     │                                     │
     │  问题 + tools 定义                 │
     ├────────────────────────────────────>│
     │                                     │
     │         选择：回答 OR 调用工具      │
     │<────────────────────────────────────┤
     │                                     │
     │                                     │
     │  如果是工具调用：                   │
     │                                     │
     │  调用真实工具                       │
     │  ┌───────────────┐                 │
     │  │  工具执行逻辑  │                 │
     │  └───────────────┘                 │
     │                                     │
     │  工具结果返回给模型                 │
     ├────────────────────────────────────>│
     │                                     │
     │         模型生成最终答案            │
     │<────────────────────────────────────┤
```

### 2.2 关键要素

| 要素 | 说明 |
|------|------|
| **tools 定义** | 用 JSON Schema 描述工具的名称、描述、参数 |
| **tool_choice** | 控制模型是否强制调用工具 |
| **tool_calls** | 模型返回的工具调用请求（JSON 格式） |
| **tool 消息** | 把工具执行结果传回模型的标准格式 |

---

## 3. How：完整实现流程

### 3.1 第一步：定义工具（JSON Schema）

```json
[
  {
    "type": "function",
    "function": {
      "name": "search_wikipedia",
      "description": "搜索维基百科获取事实信息",
      "parameters": {
        "type": "object",
        "properties": {
          "query": {
            "type": "string",
            "description": "要搜索的关键词"
          }
        },
        "required": ["query"]
      }
    }
  }
]
```

### 3.2 第二步：调用模型，传入 tools

```python
response = client.chat.completions.create(
    model="gpt-4",
    messages=messages,
    tools=tools,          # 传入工具定义
    tool_choice="auto"    # 让模型自己决定
)
```

### 3.3 第三步：判断是回答还是工具调用

| 情况 | 处理 |
|------|------|
| `message.content` 有值 | 直接返回给用户 |
| `message.tool_calls` 有值 | 执行工具，结果返回给模型 |

### 3.4 第四步：执行工具并返回结果

```python
tool_call = message.tool_calls[0]
tool_name = tool_call.function.name
tool_args = json.loads(tool_call.function.arguments)

result = execute_tool(tool_name, tool_args)

# 把结果传回模型
messages.append({
    "role": "tool",
    "content": result,
    "tool_call_id": tool_call.id
})
```

---

## 4. tool_choice 选项

| 选项 | 说明 | 适用场景 |
|------|------|---------|
| `"auto"` | 模型自己决定（默认） | 一般场景 |
| `"none"` | 不调用工具，直接回答 | 确定不需要工具时 |
| `{"type": "function", "function": {"name": "xxx"}}` | 强制调用某个工具 | 必须用这个工具时 |

---

## 5. 完整对比：Function Calling vs 其他

| 维度 | Function Calling | ReAct | ToolFormer |
|------|------------------|-------|------------|
| **实现方式** | 原生 API | Prompt | 微调 |
| **可靠性** | ✅ 高 | ⭐ 中 | ✅ 高 |
| **成本** | 低 | 低 | 高 |
| **灵活性** | ✅ 高（动态加工具） | ✅ 高 | ❌ 低（工具固定） |
| **生产就绪** | ✅ 是 | ⭐ 需打磨 | ❌ 否 |

---

## 6. 优缺点分析

### 6.1 优点
- ✅ **标准化**：JSON Schema 定义，业界标准
- ✅ **可靠性高**：模型按约定格式返回，解析稳定
- ✅ **灵活**：随时添加新工具，无需重训
- ✅ **成本低**：无需微调，直接用
- ✅ **多工具并行**：支持一次调用多个工具

### 6.2 缺点
- ❌ **依赖厂商支持**：需要模型提供 Function Calling 能力
- ❌ **Schema 编写**：需要写 JSON Schema，有点繁琐

## 7. 智能座舱场景：车辆控制示例

### 场景
用户说："把空调开到24度，再打开座椅加热。"

### Function Calling 流程

1. **模型选择工具**：调用 `set_ac_temperature(24)` 和 `turn_on_seat_heating()`
2. **执行工具**：向车机发送 CAN 指令
3. **返回结果**："空调已设为24度，座椅加热已打开"
4. **最终回答**："已为您调整好空调和座椅加热。"

## 8. 面试官追问清单

### Q1：Function Calling 的 `tool_choice` 有哪些选项？分别怎么用？
**回答要点**：
- `"auto"`：默认，模型自己决定
- `"none"`：不调用工具，直接回答
- `{"type": "function", ...}`：强制调用某个工具
- 场景：`auto` 一般用，`none` 闲聊时，强制调用在必须用工具时

### Q2：Function Calling 和 ReAct 是什么关系？
**回答要点**：
- Function Calling 是**底层 API 能力**，提供标准化的工具调用格式
- ReAct 是**高层范式**，强调"推理→行动→观察"的闭环
- ReAct 可以用 Function Calling 来实现，也可以用 Prompt 实现
- 生产环境一般用 Function Calling 实现 ReAct 范式

---

## 9. 总结提纲

1. **定义**：Function Calling 是原生 API 能力，标准 JSON 格式返回工具调用
2. **核心流程**：定义 tools → 调用模型 → 执行工具 → 结果返回
3. **tools 定义**：用 JSON Schema 描述工具的名称、描述、参数
4. **tool_choice**：auto/none/强制调用，灵活控制
5. **对比**：Function Calling 标准化、可靠、灵活，生产首选
6. **优点**：标准化、可靠、灵活、低成本、支持并行
7. **缺点**：依赖厂商支持、Schema 编写繁琐
8. **关系**：ReAct 是范式，Function Calling 是实现方式
