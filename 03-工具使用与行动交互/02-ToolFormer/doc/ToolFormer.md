# ToolFormer：让模型自己学会调用工具

## 一句话结论

ToolFormer 是 Meta 在 2023 年提出的方法，通过微调让模型**自发决定何时调用工具、调用什么工具**，无需人工编写 Prompt，在下游任务上表现出色。

---

## 1. Why：背景与痛点

### 1.1 ReAct 的局限
ReAct 虽然有效，但存在两个问题：
1. **依赖精心设计的 Prompt**：需要告诉模型"什么时候用工具"、"怎么用工具"
2. **泛化性有限**：换个任务可能需要重新写 Prompt

### 1.2 目标
有没有办法让模型**自己学会**调用工具，而不需要我们教？

---

## 2. What：ToolFormer 核心思想

### 2.1 核心流程
ToolFormer 的核心是**数据生成 + 微调**：

```
1. 构造"带工具调用"的训练数据
   ↓
2. 用这些数据微调预训练语言模型
   ↓
3. 微调后的模型会自发决定何时调用工具
```

### 2.2 关键概念
| 概念 | 说明 |
|------|------|
| **API 调用标记** | 用特殊格式表示工具调用，如 `[Calculator(2+3)]` |
| **采样** | 让模型生成候选的 API 调用 |
| **过滤** | 只保留"调用工具后能提升效果"的样本 |

---

## 3. How：数据构造（最关键）

### 3.1 三步数据构造法

```
输入文本 → 采样 API 调用 → 执行 API → 评估是否有用 → 保留/丢弃
```

### 3.2 详细步骤

| 步骤 | 说明 |
|------|------|
| **1. 采样** | 给模型输入，让它生成可能的 API 调用（如 `[Search(特斯拉)]`） |
| **2. 执行** | 真的去调用工具，拿到返回结果 |
| **3. 评估** | 对比"调用工具前"和"调用工具后"的模型预测，看效果是否提升 |
| **4. 过滤** | 只保留效果提升的样本，用于微调 |

### 3.3 评估指标
ToolFormer 使用**加权对数似然**来判断调用工具是否有用：

$$
\text{有用性} = \log P(y | x, \text{API}(x)) - \log P(y | x)
$$

如果这个值 > 0，说明调用工具后模型更有信心预测正确答案。

## 4. ToolFormer 支持的工具

ToolFormer 论文中支持了以下工具：

| 工具 | 用途 | 示例 |
|------|------|------|
| **计算器** | 数学计算 | `[Calculator(2*3+5)]` |
| **维基百科搜索** | 获取事实信息 | `[Search(特斯拉)]` |
| **问答系统** | 问答 | `[QA(首都是哪里？)]` |
| **翻译** | 机器翻译 | `[Translate(Hello, fr)]` |
| **日历** | 日期时间 | `[Calendar()]` |

## 5. ToolFormer vs ReAct 对比

| 维度 | ToolFormer | ReAct |
|------|------------|-------|
| **方式** | 微调 | Prompt 工程 |
| **是否需要 Prompt** | ❌ 不需要 | ✅ 需要 |
| **泛化性** | ✅ 强 | ⭐ 中等 |
| **成本** | 高（微调） | 低 |
| **灵活性** | 固定工具 | 可动态添加 |

---

## 6. 优缺点分析

### 6.1 优点
- ✅ **无需 Prompt**：模型自发学会调用工具
- ✅ **泛化性好**：微调后在多个任务上都能用
- ✅ **端到端**：从数据生成到微调一套流程

### 6.2 缺点
- ❌ **计算成本高**：需要生成大量数据、微调大模型
- ❌ **工具固定**：微调后很难添加新工具
- ❌ **数据构造复杂**：需要设计好的评估指标

## 7. 智能座舱场景示例

### 场景
用户问："特斯拉的续航是多少？"

### ToolFormer（微调后）
模型自发输出：
```
特斯拉的续航是[Search(特斯拉续航)]。
```
执行搜索后，模型继续输出：
```
根据搜索结果，特斯拉Model 3的续航约为600公里。
```

### 对比 ReAct
ReAct 需要这样的 Prompt：
```
你可以使用Search工具搜索信息。请用以下格式：
思考：...
行动：Search(...)
...
```

## 8. 面试官追问清单

### Q1：ToolFormer 和 ReAct 的核心区别是什么？
**回答要点**：
- ToolFormer 是**微调**方法，需要重新训练模型
- ReAct 是**Prompt 工程**方法，不需要重新训练
- ToolFormer 不需要精心设计 Prompt，模型自发学习
- ReAct 需要人工编写 Prompt，换任务可能要重写
- 成本上 ToolFormer 高但泛化性好，ReAct 灵活但依赖 Prompt

### Q2：ToolFormer 的数据构造中，"过滤"步骤是怎么做的？
**回答要点**：
- 核心是**评估调用工具是否有用**
- 用加权对数似然对比：调用工具前后的预测概率
- 如果调用工具后概率提升，说明有用，保留样本
- 如果没提升甚至下降，说明工具调用没用，丢弃
- 这样保证微调数据质量，避免噪声

## 9. 总结提纲

1. **定义**：ToolFormer 通过微调让模型自发调用工具
2. **核心**：数据生成 + 微调，无需人工 Prompt
3. **数据构造**：采样 → 执行 → 评估 → 过滤
4. **评估指标**：加权对数似然，看调用工具是否有用
5. **支持工具**：计算器、搜索、问答、翻译、日历
6. **对比 ReAct**：微调 vs Prompt，泛化 vs 灵活
7. **优点**：无需 Prompt、泛化好、端到端
8. **缺点**：成本高、工具固定、数据构造复杂
