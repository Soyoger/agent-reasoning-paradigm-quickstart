# 少样本提示：给模型"看几个例子再干活"

## 一句话结论

少样本提示（Few-shot Prompting）是在Prompt中提供少量「输入-输出」示例对，让模型快速理解任务模式并生成结果，比零样本更稳定可控，是生产环境的首选方案。

---

## 1. Why：背景与痛点

### 1.1 业务背景
- **零样本不够用**：很多任务零样本效果不稳定、格式混乱
- **需要格式控制**：JSON、SQL、代码生成等场景必须严格控制输出格式
- **快速适配新任务**：不需要收集大量标注数据，写几个示例就能上线
- **降低幻觉风险**：示例为模型提供"锚点"，减少胡编乱造

### 1.2 技术背景
- 上下文学习（In-Context Learning）：大模型不需要参数更新，仅通过Prompt中的示例就能学习任务模式
- 示例引导的概率分布偏移：示例缩小模型的生成空间，使其向目标任务对齐
- 单样本是极限情况：k=1时称为"单样本提示（One-shot）"，也是少样本的特例

### 1.3 约束与目标
| 维度 | 目标 | 约束 |
|------|------|------|
| **效果** | 稳定、可控 | 接受少量示例编写成本 |
| **成本** | 中等 | 示例数通常 1~10 个 |
| **延迟** | 可接受 | Prompt比零样本长，但仍在合理范围 |
| **格式控制** | 严格 | 必须保证输出符合预期格式 |

## 2. What：概念与边界

### 2.1 核心定义
**少样本提示**：在Prompt中提供 k（通常 1~10）个「输入-输出」示例对（Demonstrations），让模型理解任务模式后再处理新输入。

### 2.2 术语对齐
| 术语 | 说明 |
|------|------|
| **少样本 (Few-shot)** | k = 1~10 个示例 |
| **单样本 (One-shot)** | k = 1 个示例 |
| **示例 (Demonstration)** | 输入 + 期望输出的配对 |
| **上下文学习 (ICL)** | 通过Prompt中示例学习，不更新参数 |
| **零样本 (Zero-shot)** | k = 0 个示例 |

### 2.3 易混淆点澄清
| 误区 | 正确理解 |
|------|----------|
| 少样本 = 模型在学习新任务 | 不是学习，而是"激活"预训练中见过的类似模式 |
| 示例越多效果越好 | 边际效应递减，通常 3~5 个就够了 |
| 示例必须随机排序 | 顺序可能影响结果，建议固定顺序 |
| 单样本一定比少样本差 | 不一定，有时 1 个高质量示例就够了 |

### 2.4 系统边界
- **输入**：自然语言指令 + k个示例对 + 新输入
- **输出**：模型根据示例模式生成的结果
- **不包含**：思维链、工具调用（除非示例中包含）

## 3. How：原理与实践

### 3.1 原理：为什么少样本能行？

**结论句**：少样本提示通过示例在Prompt中构建"任务分布"，引导模型的生成概率向该分布对齐，本质是激活预训练知识而非重新学习。

#### 关键机制
1. **模式识别**：模型从示例中提取任务模式（输入特征→输出映射）
2. **分布对齐**：示例将模型的生成空间约束在目标任务范围内
3. **类比推理**：模型将新输入与示例进行类比，生成符合模式的输出

#### 前提假设
- 模型足够大（通常 7B+ 参数）
- 示例质量高且一致
- 任务在模型预训练分布覆盖范围内
- 示例与目标任务分布一致

#### 复杂度分析
| 维度 | 分析 |
|------|------|
| **Prompt长度** | O(指令长度 + k×示例长度) |
| **推理时间** | O(输出长度)，略长于零样本 |
| **内存占用** | 中等 |
| **实现难度** | 中等（需要编写示例） |

---

### 3.2 流程：少样本提示的标准步骤

```
1. 明确任务目标
   ↓
2. 收集/编写示例
   ├─ 覆盖常见情况
   ├─ 覆盖边界情况
   ├─ 示例格式一致
   └─ 示例质量高
   ↓
3. 构造Prompt
   ├─ 指令
   ├─ 示例1
   ├─ 示例2
   ├─ ...
   └─ 新输入
   ↓
4. 调用模型获取输出
   ↓
5. 验证并后处理（可选）
```

### 3.4 技术选型：零样本 vs 少样本 vs 单样本

| 维度 | 零样本 | 单样本 | 少样本（3-5个） |
|------|--------|--------|----------------|
| **示例数** | 0 | 1 | 3-5 |
| **优点** | • Prompt最短<br>• 延迟最低<br>• 实现最简单 | • 比零样本稳定<br>• 格式控制更好<br>• 成本可控 | • 效果最稳定<br>• 格式控制严格<br>• 覆盖边界情况 |
| **缺点** | • 效果不稳定<br>• 格式控制弱<br>• 容易幻觉 | • 效果不如少样本<br>• 可能过拟合单个示例 | • Prompt较长<br>• 需要写多个示例<br>• 边际效应递减 |
| **成本** | 最低 | 低 | 中等 |
| **复杂度** | 最低 | 低 | 中等 |
| **适用场景** | • 简单分类<br>• 快速原型<br>• 摘要生成 | • 格式控制<br>• 中等复杂度任务<br>• 快速迭代 | • 严格格式要求<br>• 生产环境<br>• 高精度要求 |

**选择依据**：
- 先试零样本，不行上单样本，还不行上少样本
- 格式要求严格 → 至少单样本
- 生产环境 → 优先少样本（3-5个）
- 成本敏感 → 单样本可能是最佳平衡点
- 效果优先 → 少样本（3-5个）

### 3.5 实现要点：写出高质量少样本Prompt

#### 要点1：示例格式一致
```
❌ 不好：
用户："我要听周杰伦的歌"
答：音乐播放

用户："导航到公司"
输出：导航设置

✅ 好：
用户："我要听周杰伦的歌"
输出：音乐播放

用户："导航到公司"
输出：导航设置
```

#### 要点2：覆盖常见和边界情况
```
✅ 好的示例覆盖：
1. 常见情况："我要听周杰伦的歌" → 音乐播放
2. 边界情况："打开车窗和空调" → 多意图处理
3. 异常情况："你好" → 闲聊
```

#### 要点3：示例顺序有讲究
```
✅ 推荐：按逻辑顺序排列，或固定随机种子
✅ 也可以：将最相关的示例放在最后（近因效应）
```

#### 要点4：示例质量 > 数量
```
✅ 3个高质量示例 > 10个低质量示例
高质量示例：明确、一致、有代表性
```

#### 生产级Prompt模板
```markdown
你是一个{角色}。
请{任务描述}。

示例：
{示例1}
{示例2}
{示例3}

新输入：{输入内容}

输出：
```

## 4. 优化与改进方案

### 方案1：示例选择优化（工程优化）

**收益**：准确率提升 10%~20%  
**代价**：需要额外逻辑选择示例  
**适用条件**：有大量历史数据的场景

**具体做法**：
- 语义相似度选择：选与新输入最相似的历史示例
- 多样性选择：覆盖不同类型的示例
- 最近邻选择：从向量数据库检索最相关示例

**验证方式**：A/B测试，对比准确率

---

### 方案2：示例压缩（工程优化）

**收益**：Token成本降低 20%~40%  
**代价**：可能损失部分信息  
**适用条件**：Token成本敏感的场景

**具体做法**：
- 简化示例描述，保留核心信息
- 使用更简洁的格式
- 合并相似示例

**验证方式**：对比效果变化 + 成本监控

---

### 方案3：动态示例数（架构优化）

**收益**：兼顾效果和成本  
**代价**：系统复杂度增加  
**适用条件**：对效果和成本都有要求的生产系统

**架构设计**：
```
请求 → 零样本 → 置信度检查
         ↓
    置信度够？
     ↓    ↓
    是    否 → 单样本 → 置信度检查
                      ↓
                 置信度够？
                  ↓    ↓
                 是    否 → 少样本 → 输出
                  ↓
                输出
```

**验证方式**：端到端效果 + 成本监控

## 5. 适用场景分析

### 5.1 推荐使用场景

| 场景类型 | 具体例子 | 为什么适合 |
|---------|---------|-----------|
| **严格格式输出** | JSON、SQL、代码生成 | 示例保证输出格式正确 |
| **复杂分类** | 多标签分类、层次分类 | 示例帮助理解分类边界 |
| **结构化输出** | 表格提取、实体关系抽取 | 示例明确输出结构 |
| **生产环境** | 核心业务链路 | 效果稳定可控，降低风险 |
| **新任务适配** | 快速上线新功能 | 几个示例就能启动 |

**结论句**：少样本是生产环境的首选，在效果、成本、稳定性之间找到了最佳平衡点。

---

### 5.2 不推荐使用场景

| 场景类型 | 具体例子 | 为什么不适合 |
|---------|---------|-------------|
| **超复杂推理** | 数学证明、复杂逻辑题 | 需要思维链，光有示例不够 |
| **海量知识依赖** | 百科全书级问答 | 需要RAG，示例装不下那么多知识 |
| **Token极端敏感** | 边缘设备、超低预算 | 零样本更划算 |
| **任务太简单** | 二选一的简单分类 | 零样本可能就够了 |

## 7. 智能座舱场景落地示例

### 场景：座舱意图少样本识别

**映射到 Why**：
- 痛点：零样本格式不稳定，意图识别准确率不够
- 约束：需要严格控制输出格式，准确率要求高
- 目标：稳定、准确的意图识别

**映射到 What**：
- 输入：用户语音转文字
- 输出：意图类别（带格式约束）
- 提供3个示例

**映射到 How**：

#### Prompt 设计
```markdown
你是智能座舱的专业意图识别助手。
请将用户请求分类到以下类别之一：
[音乐播放, 导航设置, 空调控制, 电话拨打, 车窗控制, 音量调节]
仅输出类别名称，不要解释。

示例：
用户："我要听周杰伦的歌"
输出：音乐播放

用户："导航到上海迪士尼"
输出：导航设置

用户："把空调调到24度"
输出：空调控制

新输入："开一下车窗"

输出：
```

#### 效果验证
| 指标 | 零样本 | 少样本(3个) |
|------|--------|-------------|
| 准确率 | 82% | 95% |
| 格式错误率 | 8% | 0% |
| 平均延迟 | 110ms | 130ms |
| Prompt Token数 | 87 | 215 |

#### 收益
- 准确率提升 13%
- 格式错误率降为 0
- 延迟仅增加 18%，完全可接受

## 8. 面试官追问清单

### Q1：少样本提示中，示例数量多少合适？
**回答要点**：
- 不是越多越好，边际效应递减
- 通常 3~5 个就够了
- 先试 1 个（单样本），不行再加
- 要看任务复杂度和示例质量
- 可以做 ablation study 确定最佳数量

### Q2：示例顺序会影响结果吗？怎么处理？
**回答要点**：
- 会，有研究表明顺序有影响
- 处理方式：
  1. 固定一个顺序，不要随便变
  2. 可以尝试将最相关的示例放最后（近因效应）
  3. 做A/B测试，看哪个顺序效果好
  4. 生产环境固定下来，不要频繁调整

### Q3：如何写高质量的少样本示例？
**回答要点**：
- 格式一致：所有示例用同样的格式
- 覆盖全面：常见情况 + 边界情况 + 异常情况
- 质量高 > 数量多：3个高质量示例 > 10个低质量示例
- 明确：每个示例的输入输出都清晰，没有歧义
- 代表性：每个示例代表一种典型情况

### Q4：少样本提示和微调（Fine-tuning）怎么选？
**回答要点**：
- 少样本：
  - 优点：快速、无需数据、无需训练
  - 缺点：效果有上限、Prompt长
  - 适用：快速迭代、数据少、中小规模
- 微调：
  - 优点：效果更好、Prompt短、推理快
  - 缺点：需要数据、需要训练、成本高
  - 适用：大规模生产、数据充足、效果要求极高
- 选择：先少样本，效果不够再考虑微调

### Q5：少样本提示会过拟合吗？怎么避免？
**回答要点**：
- 会，可能过拟合到示例的特殊模式
- 避免：
  1. 示例多样化，不要都太相似
  2. 覆盖边界情况，不要只给简单示例
  3. 不要用太多示例（边际效应递减+过拟合风险）
  4. 定期评估，发现过拟合及时调整示例
  5. 可以做示例消融测试，看哪些示例是关键的

## 9. 总结提纲（面试背诵版）

1. **定义**：少样本提示 = 指令 + k个示例对 + 新输入，k通常1~10
2. **本质**：通过示例构建任务分布，引导模型生成，不是重新学习
3. **优势**：效果稳定、格式控制强、幻觉少、快速适配新任务
4. **劣势**：Prompt较长、成本中等、需要写示例
5. **适用**：严格格式输出、复杂分类、生产环境、新任务快速上线
6. **不适用**：超复杂推理、海量知识依赖、Token极端敏感
7. **选型**：零样本 → 单样本 → 少样本，逐步升级
8. **示例**：格式一致、覆盖全面、质量>数量、3~5个通常够
9. **优化**：示例选择、示例压缩、动态示例数
10. **生产**：少样本是生产首选，效果成本稳定性最佳平衡
